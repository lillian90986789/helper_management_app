# 临时任务（图片描述） — Design

## 背景

现在 `TaskNew.jsx` 只支持一种建任务方式：按"每周哪几天重复"的模板（`TaskTemplate` → 每天生成一条 `DailyTask`）。雇主没法快速派发一次性、临时的活（例如"把阳台这个箱子搬开"），只能生硬地套用周重复模板，或者口头交代。

## 目标

新增"临时任务"创建方式：雇主拍照/上传参考图 + 文字说明，指定执行人，即可派发一个不需要每周重复的一次性任务。

## 非目标

- 不支持"重复"属性（临时任务本身就是一次性的，不需要周几选择器）。
- 不引入草稿/审批发布流程，创建即直接派给执行人（与现有模板任务发布逻辑一致）。
- 不改变任务执行侧（女佣端）的完成/提交/审核流程。

## 设计

### 1. 入口与交互

`TaskNew.jsx` 页面顶部新增一个 tab 切换：**重复任务 / 临时任务**。

- 「重复任务」= 现有表单不变（周几选择器 + 模板逻辑）。
- 「临时任务」= 简化表单，去掉周几选择器，新增图片上传区；提交后直接创建一条一次性任务，不写入 `TaskTemplate`。

### 2. 临时任务表单字段

| 字段 | 说明 |
|---|---|
| 标题 | 必填，同现有 `taskName` |
| 参考图片 | 支持多张上传（多角度拍照），至少 0 张（非强制，但一般会传） |
| 文字说明 | 同现有 `description`，可选 |
| 执行人 | 必填，同现有 assignee 选择（家庭内女佣/成员） |
| 优先级 | 同现有 normal/important/urgent |
| 截止日期 | 可选，不选默认视为"今天" |
| 区域 | 必填，沿用现有区域选择器（与模板任务一致，保证任务统计/按区域筛选不遗漏临时任务） |
| 完成要求 | 沿用现有 Toggle：`require_photo`（完成需拍照）、`require_approval`（需审批） |

### 3. 数据模型

复用现有 `DailyTask` 表，不新建表：

- 创建一条 `DailyTask`，`task_template_id = NULL`（标记这是一次性任务，非模板生成），`task_date` = 选择的截止日期（默认今天），其余字段（`assignee_id`/`priority`/`require_photo`/`require_approval`/`description_snapshot` 等）直接填入，`status` 初始为 `today_todo`（与模板生成的每日任务一致，进入相同的执行/审核状态机）。
- 雇主上传的参考图片写入 `DailyTaskAttachment`（`daily_task_id` 指向新建的 `DailyTask`，`uploader_id` = 雇主用户 id，`file_type` 用一个新的值如 `'reference'` 区分于女佣提交完成证明用的默认 `'image'`）。
- 女佣端任务详情页展示时，需要按 `file_type` 区分"雇主发的参考图"和"我提交的完成照片"两组，不要混在一起显示。

### 4. 后端改动

- 新增一个创建接口（或复用/扩展现有创建 `DailyTask` 的接口），入参包含 `task_date`、`assignee_id`、`priority`、`description`、`require_photo`、`require_approval`、`area_id`，以及参考图片 URL 列表（先走现有图片上传接口拿到 URL，再随表单一起提交，与现有"完成时传图"走同一套上传接口）。
- 列表查询（`GET /daily-tasks` 或对应聚合接口）本身按 `task_date` 过滤即可覆盖临时任务，因为它复用同一张表，不需要额外的列表接口。

## 测试关注点

- `task_template_id = NULL` 的任务在任务列表/统计（如"今日待办数"）里要能被正常统计，不能因为没有模板关联而被漏掉或报错。
- 参考图片与完成图片的 `file_type` 区分要在前端详情页正确分组展示，避免雇主的参考图被误当成女佣的完成证明。
- 截止日期早于今天/远晚于本周的边界情况不做限制，但需确认前端日期选择器体验合理。
